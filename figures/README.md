# Figures and Visualizations Directory

This directory contains the plotted results, cache summaries, and profiling galleries generated by the analysis script `scripts/generate_visualizations.py` from the raw benchmark outputs.

---

## 1. Overview of Artifacts

The following files are located in this directory (or generated dynamically here):

| File Name | Format | Description |
| :--- | :--- | :--- |
| **`throughput.png`** | Image (PNG) | Comparison plot of message transfer rates (GB/s) versus payload sizes. |
| **`latency.png`** | Image (PNG) | Comparison plot of message latency (microseconds) versus payload sizes, plotted on a logarithmic scale. |
| **`speedup.png`** | Image (PNG) | Line chart showing the throughput speedup factor of `io_uring` compared to the other three baselines. |
| **`cache_misses.png`** | Image (PNG) | Bar charts comparing L1 Data Cache and Last-Level Cache (LLC) miss rates. |
| **`cache_misses_summary.csv`** | CSV Data | Structured table containing raw counts and rates of cache loads/misses. |
| **`flamegraphs/`** | Directory | Contains the interactive profiling flamegraph SVG assets and the HTML gallery index. |
| **`summary.md`** | Markdown | High-level summary log of the generation run. |

## 2. Interpretation of the Visualizations

### 2.1 Throughput Plot (`throughput.png`)
- **What it shows**: On the X-axis, the message size sweeps from 64 bytes to 1 MiB. On the Y-axis, it shows the media throughput achieved in Gigabytes per second (using the GiB/s convention in calculation).
- **Core Insights**: 
  - For small message sizes (64 B - 4 KiB), throughput is dominated by synchronization overhead and system call frequency.
  - As payload sizes scale up to 1 MiB, the throughput of all implementations increases.
  - The `io_uring` Shared Ring implementation peaks much higher than the others because it bypasses kernel copying and context-switching overhead in the hot path.

### 2.2 Latency Plot (`latency.png`)
- **What it shows**: Measures end-to-end transfer latency in microseconds. It utilizes a logarithmic Y-axis to cover values ranging from sub-microsecond levels (for small memory ring handoffs) up to hundreds of milliseconds (for large message transfers on constrained transports).
- **Core Insights**:
  - The Interquartile Range (IQR) bands (drawn as light transparent colors around the lines) show run-to-run consistency. Narrower bands denote high predictability.
  - POSIX message queues (`mqueue`) and Sockets typically show higher latencies due to kernel scheduling and queue lock transitions.
  - `io_uring` maintains the lowest latency across the scale due to lock-free atomic SPSC indexing.

### 2.3 Speedup Plot (`speedup.png`)
- **What it shows**: The throughput ratio:
  $$\text{Throughput Ratio} = \frac{\text{Throughput of io\_uring}}{\text{Throughput of Baseline IPC}}$$
  A ratio of $1.0$ is plotted as a horizontal dashed line representing parity. Ratios above $1.0$ show the speedup factor of the shared-ring mechanism.
- **Core Insights**: 
  - Visualizes the massive advantage of lock-free memory mapping at medium payload sizes.
  - Compares where the performance curves diverge most heavily, highlighting cache-sensitive sizing sweet spots.

### 2.4 Cache Misses Plot (`cache_misses.png`)
- **What it shows**: Side-by-side bar charts displaying L1 data cache load-miss rates and LLC load-miss rates.
- **Core Insights**:
  - Informs whether memory bandwidth or cache thrashing is a performance bottleneck.
  - Shows that `io_uring` maintains high cache locality due to cache-line alignment (`alignas(64)`) on its ring slots, reducing LLC misses and false sharing.

## 3. Detailed Explanation of the Flamegraph Profiles

The flamegraph SVGs located in the `flamegraphs/` subdirectory illustrate the distribution of CPU execution time across user-space applications, standard library interfaces, and kernel subsystems. Below is a comprehensive breakdown of the characteristics, call-stack hotspots, and behaviors represented in each profile:

### 3.1 POSIX Pipe Profile (`pipe_flamegraph.svg`)
- **Visual Pattern**: Broad stacks centered around blocking reads and writes, indicating high synchronization frequency and CPU cores repeatedly yielding execution.
- **Key Call Stacks & Hotspots**:
  - **`pipe_write` & `pipe_read`**: The core kernel file operations responsible for managing the FIFO circular buffer. They occupy a wide horizontal band, representing the time spent locking the pipe mutex, managing queue bounds, and copying memory.
  - **`copy_user_enhanced_fast_string` / `copy_to_user` / `copy_from_user`**: These frames expand significantly as message sizes increase, illustrating the overhead of moving memory between user space and kernel space buffers.
  - **`autoremove_wake_function` & `try_to_wake_up`**: Stacks associated with scheduler activity. These frames become prominent because the producer and consumer frequently wait on full or empty buffer conditions, triggering context switches and scheduling wakeups.

### 3.2 Unix Domain Socket Profile (`socket_flamegraph.svg`)
- **Visual Pattern**: Deep, complex call hierarchies. This profile shows the highest proportion of kernel execution overhead due to socket buffer management and connection framing.
- **Key Call Stacks & Hotspots**:
  - **`unix_stream_sendmsg` & `unix_stream_recvmsg`**: Main entry points for UNIX stream sockets. They encompass extensive packetization, flow control checks, and socket lock routines.
  - **`sock_alloc_send_pskb` & `skb_copy_datagram_from_iter`**: Frame allocations for socket descriptor buffers (`sk_buff`). The CPU spends considerable cycles allocating, initializing, and copying data blocks inside these kernel packet structures.
  - **`mutex_lock` & `spin_lock`**: Frequent lock contention is visible, as socket read/write operations must serialize access to the connection queues, increasing CPU wait states under stress.

### 3.3 POSIX Message Queue Profile (`mq_flamegraph.svg`)
- **Visual Pattern**: Flat, kernel-heavy profile reflecting message-oriented IPC constraints and virtual filesystem (VFS) operations.
- **Key Call Stacks & Hotspots**:
  - **`sys_mq_timedsend` & `sys_mq_timedreceive`**: The primary system calls for message dispatch. They handle POSIX queue serialization, priority checks, and state validation.
  - **`pipelined_send` & `pipelined_receive`**: A specialized kernel optimization path. When a receiver is already waiting, the kernel bypasses the intermediate queue buffer and copies the message payload directly into the consumer's memory buffer, showing up as a wide memory-copy frame.
  - **Virtual Filesystem (VFS) Overheads**: Since POSIX message queues are represented as nodes under the `mqueue` virtual filesystem, file permission checks, descriptor allocation, and inode lock modifications (`mqueue_evict_inode`) occupy a visible portion of the stack.

### 3.4 io_uring Shared Ring Profile (`uring_flamegraph.svg`)
- **Visual Pattern**: Extremely shallow userspace stacks for the benchmark processes, contrasted with a separate, highly focused kernel thread polling stack.
- **Key Call Stacks & Hotspots**:
  - **Lock-Free Userspace Spinloops**: The active producer and consumer processes (`uring_producer` and `uring_consumer`) spend almost their entire CPU budget executing userspace loops checking atomic index pointers. This is represented by wide, shallow frames compiling directly to `CPU_PAUSE` instructions (`_mm_pause` or assembly `yield`).
  - **`io_sq_thread` (Kernel Thread)**: When SQPOLL is active, the kernel thread `io_sq_thread` executes on Core 3. The flamegraph captures this thread executing `io_submit_sqes` and completing NOP entries. This architecture completely isolates kernel task submissions onto Core 3, preventing system-call transitions on Core 1 and Core 2.
  - **`io_uring_enter` (Fallback Mode)**: In environments where SQPOLL is inactive, system-call entries appear under `io_uring_enter`. However, these remain narrow compared to sockets, as the payload transfer itself bypasses the kernel, relying purely on the mapped shared memory slots (`mmap`).

## 4. How to View the Flamegraph Gallery

The folder `flamegraphs/` contains `index.html`. This page embeds the SVGs generated during profiling. 

To view it:
1. Open a terminal and navigate to this repository.
2. Open `figures/flamegraphs/index.html` in your web browser:
   ```bash
   # On macOS:
   open figures/flamegraphs/index.html
   
   # On Linux:
   xdg-open figures/flamegraphs/index.html
   ```
3. You can click on individual frames in the SVGs to zoom into stack traces or search for specific function names (e.g., `write`, `copy_user`, `io_sq_thread`).
